## Observación 1 — `Storage::S3#s3_client` debe verificarse

**Hallazgo:** El plan asume que `s3_client` es un private method existente en `Storage::S3`. Pero no hay garantía de que exista o de su signatura.

**Verificar:**
```bash
grep -n "def s3_client" lib/data_drain/storage/s3.rb
```

**Si no existe:** implementar con:
```ruby
def s3_client
  @s3_client ||= Aws::S3::Client.new(region: configuration.aws_region)
end
```

**Si existe con firma diferente:** ajustar en la implementación.

## Observación 4 — Path separators en Windows

**Hallazgo:** El plan usa `File.join` y `File.basename` que manejan correctamente `\` y `/` en Ruby. Pero en Windows el `s3_key` resultante podría tener backslashes.

**Mitigación ya en el plan:**
```ruby
s3_key = "#{folder.chomp("/")}/#{actual_filename}"
```

Pero si `actual_filename` viene de `File.basename(local_path)` en Windows, podría tener backslashes. Ruby los normaliza, pero el test debería verificar esto.

**Test sugerido:**
```ruby
it "maneja paths de Windows correctamente" do
  # En sistemas Windows, File.basename("C:\\scripts\\export.py") => "export.py"
  # El s3_key resultante debería ser "scripts/export.py", no "scripts\\export.py"
end
```

## Observación 6 — `Storage::Local#upload_file` es no-op en tests de integración

**Hallazgo:** Si un servicio usa `storage_mode = :local` en tests (sin mocks de S3), `upload_script` levantará `ConfigurationError`. Esto es correcto según el plan.

**Pero:** ¿debería `Storage::Local#upload_file` estar implementado para tests locales reales? El plan lo incluye, lo cual está bien.

**Verificar:** Que los tests de integración de GlueRunner con `storage_mode = :local` fallen temprano con mensaje claro.

## Observación 7 — Content-Type para `.py` podría mejorarse

**Hallazgo:** El plan usa `text/x-python` explícito. AWS S3 infiere correctamente `.py` como `text/x-python` si no se provee.

**Alternativa:** Hacer `content_type` opcional y dejar que S3 infiera:
```ruby
params[:content_type] = content_type if content_type
```

Pero el plan usa explícito, lo cual está bien (más predictible).

## Observación 8 — No hay test de concurrencia

**Hallazgo:** Si dos procesos corren `upload_script` con el mismo destino simultáneamente, el último gana. No hay lock.

**Recomendación:** Documentar como limitación en Fase 6. Para v0.5.0 es aceptable (scripts son pequeños y el riesgo es bajo).
